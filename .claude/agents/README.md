# Agent roles for dotkeeper

This directory defines the roles agents take when working on dotkeeper.
Each file is a "role card" — a brief that constrains scope, forbids
out-of-role actions, names the preferred model tier, and specifies
escalation rules to hand work off to another role.

These are used by Claude Code's subagent feature:

```
Task(subagent_type="dotkeeper-docs", prompt="review PR #123 for doc impact")
```

The subagent picks up the role card as its system brief and operates
within its defined scope.

## The six roles

| Role | Primary model | Typical work |
|---|---|---|
| [dotkeeper-arch](dotkeeper-arch.md) | Opus 4.7 | Architectural decisions, ADRs, cross-cutting refactor planning |
| [dotkeeper-impl](dotkeeper-impl.md) | Sonnet 4.6 | Go feature work and its unit tests, per ADR specs |
| [dotkeeper-qa](dotkeeper-qa.md) | Sonnet 4.6 | Test coverage, integration tests, PR review for test adequacy |
| [dotkeeper-docs](dotkeeper-docs.md) | Sonnet 4.6 | README, architecture docs, migration guides, in-code WHY comments |
| [dotkeeper-release](dotkeeper-release.md) | Haiku 4.5 | Version bumps, packaging (AUR, brew, nixpkgs), changelog, release notes |
| [dotkeeper-triage](dotkeeper-triage.md) | Sonnet 4.6 + Gemini for long-context reads | Issue intake, CI-failure investigation, localized fixes or escalation |

## Model-tier policy

- **Opus 4.7** — architectural judgment only. Aim for under 10% of total
  token spend on dotkeeper work.
- **Sonnet 4.6** — the default workhorse for substantive scoped work
  (implementation, testing, documentation, triage). 70–80% of spend.
- **Haiku 4.5** — mechanical tasks (version bumps, packaging, lint
  fixes, routine doc boilerplate). 10–15% of spend.
- **Gemini CLI** — long-context whole-repo reads during triage or audit.

Reach for Sonnet by default; escalate to Opus only when judgment beyond
a spec is required; drop to Haiku when the task is purely mechanical.

## Writing rules that apply across all roles

These are not repeated in every role card:

- No AI attribution in commits (no `Co-Authored-By: Claude`, no
  "Generated by ..."). See the repo's `.agent/AGENT.md`.
- PR-based workflow, not direct-to-main. See the repo's `.agent/AGENT.md`.
- `make build`, `make test`, `go vet -tags noassets ./...` must pass
  before any PR merges.
- Comments: write WHY, not WHAT. Well-named code tells you what;
  comments earn their place by explaining constraints, invariants, or
  surprises.
