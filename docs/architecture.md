# dotkeeper v0.6 Architecture

A walkthrough of the current design decisions behind dotkeeper. The ADRs are
the durable decision record; this document is the readable narrative.

## At a glance

- dotkeeper syncs repos and dotfiles across machines with embedded Syncthing,
  plus staggered git backup for history and rollback.
- Desired state is declarative. A reconciler loop continuously compares desired
  config with observed Syncthing/Git/filesystem state and applies the diff.
- Machine-level config lives in `machine.toml`; tool-owned runtime state lives
  in `state.toml`.
- Per-repo desired state lives inside each local repo copy as `.dotkeeper.toml`.
  That file is local machine state. It is ignored by Git and Syncthing.
- Public repos stay clean: dotkeeper's own repo contains product code and docs,
  not private topology or local sync markers.

## The reconciler model

dotkeeper behaves like a small local controller:

```text
reconcile():
    desired  = read_config()      # machine.toml + local .dotkeeper.toml files
    observed = query_state()      # Syncthing REST API + git + filesystem
    actions  = diff(desired, observed)
    for action in actions:
        apply(action)
    record_observation(observed)
```

`diff()` is pure: same inputs, same action list. `apply()` is where side
effects happen, and each action is idempotent. That makes reconcile safe to run
from file events, a timer, and explicit `dotkeeper reconcile` calls.

The daemon triggers reconcile when:

| Trigger | When |
|---------|------|
| fsnotify watcher | `machine.toml`, `state.toml`, or local `.dotkeeper.toml` changes |
| Periodic timer | Every `machine.reconcile_interval` (default 5 min) |
| `dotkeeper reconcile` | Manual one-shot, blocks until complete |

## State locations

### `machine.toml`

Location: `$XDG_CONFIG_HOME/dotkeeper/machine.toml`
(usually `~/.config/dotkeeper/machine.toml`)

This is declarative, human- or Nix-authorable machine identity and policy:

- machine name and backup slot
- default share targets and backup intervals
- declared peers
- scan roots and exclusions

It has no secrets and can be generated by Home Manager.

### `state.toml`

Location: `$XDG_STATE_HOME/dotkeeper/state.toml`
(usually `~/.local/state/dotkeeper/state.toml`)

This is tool-owned runtime state:

- this machine's Syncthing device ID
- imperative peer entries from `dotkeeper peer add`
- `tracked_overrides` from `dotkeeper track`
- cached observed repo and peer state

Users should not hand-edit this file.

### `.dotkeeper.toml`

Location: `<repo>/.dotkeeper.toml`

This is local desired state for one local repo copy. Its presence opts that
copy into dotkeeper management. Its absence means dotkeeper ignores the repo,
even if the repo lives under a scan root.

The file is deliberately local-only:

- `dotkeeper track <path>` writes it when missing.
- `dotkeeper track` and reconcile ensure `.stignore` excludes it from
  Syncthing.
- dotkeeper also writes `.git/info/exclude` entries so `.dotkeeper.toml`,
  `.stignore`, `.dkfolder`, `.stfolder`, Syncthing temp files, and conflict
  artifacts do not enter Git by accident.
- `dotkeeper.toml` is not read as a repo config file.

Nix/Home Manager can materialize `.dotkeeper.toml` on each machine from private
flake data. Non-Nix machines can use `dotkeeper track`.

## Discovery

dotkeeper finds managed repos from three sources:

1. **Scan-root walk.** `machine.toml` declares roots under `[discovery]`.
   dotkeeper walks them to the configured depth and looks for
   `<dir>/.dotkeeper.toml`.
2. **Tracked overrides.** `dotkeeper track <path>` records explicit paths in
   `state.toml` so repos outside scan roots can still be reconciled.
3. **Observed Syncthing folders.** Existing Syncthing folders are observed so
   drift can be detected and corrected, but desired state still comes from the
   local config files above.

The two consent boundaries are deliberate: a directory must be under a declared
scan root or explicit tracked override, and the repo must have local
`.dotkeeper.toml`.

## What does not sync

Every dotkeeper-managed repo gets a managed `.stignore` with baseline excludes
for:

- `.git`
- `.dotkeeper.toml`
- `dotkeeper.toml`
- `.stignore`, `.dkfolder`, `.stfolder`
- `.syncthing.*.tmp`
- `*.sync-conflict-*`
- common dependency caches and build outputs such as `node_modules`, `dist`,
  `build`, `.next`, `.cache`, `.venv`, and `target`

This is the hotfix boundary that prevents sync conflict storms in Git internals,
build directories, dependency caches, and dotkeeper's own local markers.

## Commands

| Command | Purpose |
|---------|---------|
| `dotkeeper init` | Generate Syncthing identity, write `state.toml`, scaffold `machine.toml` |
| `dotkeeper start` | Run the daemon |
| `dotkeeper reconcile` | Force one reconcile pass now |
| `dotkeeper identity` | Print machine name and Syncthing device ID |
| `dotkeeper peer add/list/remove` | Manage imperative peers in `state.toml` |
| `dotkeeper track <path>` | Bootstrap local `.dotkeeper.toml`, local excludes, and tracked override state |
| `dotkeeper untrack <path>` | Remove a tracked override |
| `dotkeeper doctor` | Run health checks |
| `dotkeeper conflict list` | List pending Syncthing conflict files |
| `dotkeeper conflict resolve-all` | Batch auto-resolve trivial conflicts |

## Contributor map

| Package | Role |
|---------|------|
| `internal/config/` | Read/write `machine.toml`, `state.toml`, and `.dotkeeper.toml` |
| `internal/discovery/` | Shared scan-root discovery for managed repos |
| `internal/reconcile/` | Desired/observed model, `Diff`, action types, and applier |
| `internal/stengine/` | Embedded Syncthing lifecycle |
| `internal/stclient/` | Syncthing REST API client |
| `internal/conflict/` | Sync-conflict detection and resolution |
| `internal/doctor/` | `dotkeeper doctor` health checks |

To add a reconcilable property, add it to the config model, add an action in
`internal/reconcile/`, implement the apply path, and pin the behavior with
focused tests.
