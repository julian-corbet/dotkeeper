# dotkeeper — per-repo agent instructions

Overrides and extensions to the global `~/.agent/AGENT.md`. Load this when working in this repo.

## Goal

dotkeeper is a FOSS tool — AGPL-3.0 licensed, public on GitHub since 2026-04-19. Every change lands in a publicly-visible review trail.

## Workflow: PR-based, not direct-to-main

**Override of the global "commit and push directly to `main`" rule.**

All non-trivial changes to `main` must go through a pull request:

1. Create a feature branch: `git checkout -b <topic>/<short-name>`
2. Commit work on the branch.
3. Open a PR: `gh pr create --title "..." --body "..."`
4. Merge with `gh pr merge --squash --delete-branch` once checks pass.

Exceptions (still fine to push directly):
- Dependabot auto-opens PRs already; they're merged normally.
- Auto-commits from dotkeeper's own git timer (`auto: <machine> ...`).
- Release tags — `git tag` + `git push origin <tag>` is not a commit.
- Emergency rollbacks when CI is broken and blocking everything.

**Enforced mechanically** via branch protection on `main`: PR required, `build` and `lint` status checks required, linear history required, no force-pushes, no deletions. Direct-to-main `git push` will be rejected by the remote.

## Commit messages

Still no AI attribution. No `Co-Authored-By: Claude`, no "Generated by ...". Standard conventional-ish commit messages.

## Testing gate before merging any PR

- `make build` passes
- `make test` passes
- `go vet -tags noassets ./...` clean
- CI workflow green
- For changes to `internal/stengine`, manual smoke test: rebuild, restart service, confirm `dotkeeper status` still peers. Syncthing's lifecycle is finicky.

## Branch protection on main

**Active since 2026-04-19.** Settings:
- Require PR before merging; no approvals required (solo project, self-merge fine)
- Required status checks: `build`, `lint`
- Linear history required (matches squash-merge convention)
- No force pushes, no deletions
- `enforce_admins: false` (admin can recover from wedged states without a self-issued exception)

If settings need adjusting, the config lives at `github.com/julian-corbet/dotkeeper/settings/branches` (UI) or via `gh api repos/julian-corbet/dotkeeper/branches/main/protection` (API). Restore-from-scratch JSON is in the v0.1.3+ commit history of this file if needed.

## Scope discipline

Do **not** preemptively rewrite significant portions for elegance. dotkeeper is small and works. Refactors need a specific trigger (bug, feature, CVE, clarity problem on a hot path). Cosmetic churn loses review credibility.

## Dotkeeper-specific knowledge

- **Syncthing ClientName** is hardcoded upstream (`lib/connections/service.go:312`) and cannot be overridden without forking. `ClientVersion` works via `build.Version` exported var. We accept the "syncthing" ClientName on the wire.
- **`auto.Assets` / `lib/api`** is unconditionally pulled via `stlib.App.startup → setupGUI`. Naked `go install` fails. README documents the `-tags noassets` workaround. Revisit if Syncthing restructures.
- **Log redirection** uses fd-level `unix.Dup2` (not `syscall.Dup2` — missing on riscv64/arm64/386). Unix-only; Windows has a no-op stub.
- **Staging directories** in the release workflow are named `dotkeeper_<version>_<suffix>/`. The version dots caused `dotkeeper_*.*` globs to match directories. Upload-artifact globs must enumerate extensions explicitly.
- **nFPM install** — upstream renames their asset filenames between releases. Pin the version and the exact filename shape in release.yml.
